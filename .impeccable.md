# .impeccable.md

## Design Context

### Users
Solo developers and small teams using AI coding agents (Claude Code, Cursor, Copilot) as their primary workflow. They're already in the terminal or editor — the web UI is a glanceable board, not their primary input. They're technical, efficient, and intolerant of noise. The job: understand project state at a glance, read ticket details, and make occasional edits.

### Brand Personality
**Calm · Focused · Reliable**

The interface should feel like a well-made tool that gets out of the way. No delight for its own sake. No personality injected where it doesn't belong. Trust is built through consistency and restraint.

### Aesthetic Direction
- **Reference**: Linear — polished, dense, dark-first, keyboard-aware, premium dev tool feel. Emulate the information density and precision, not the flashiness.
- **Anti-reference**: Notion/Jira — don't be bloated, don't be bubbly, don't pad everything.
- **Direction**: Minimal — low chrome, let content breathe. One accent colour does all the interactive work. Monospace for ticket IDs and code. Near-zero decorative elements.
- **Colour**: Neutral grey palette (near-black/near-white base) + one muted accent (violet or indigo). Status colours are semantic and muted — never loud.
- **Theme**: Both light and dark via CSS variable tokens. No `dark:` prefix — switching toggles a root attribute that swaps the token values.

### Design Principles

1. **Content is the UI** — the kanban board and ticket details are the product. Every pixel of chrome must justify its existence.
2. **One accent, used sparingly** — interactive elements, focus rings, and active states use the accent. Everything structural uses neutrals.
3. **Density with breathing room** — pack information like Linear, but respect whitespace at card and panel boundaries.
4. **Calm motion** — transitions exist only to preserve spatial context (e.g. drawer sliding in). No bounce, no spin, no gratuitous animation.
5. **Status communicates clearly** — each ticket status has a distinct, muted colour and icon. Scannable at a glance without shouting.

### Token Architecture

All colours are CSS variables on `:root` (light) and `[data-theme="dark"]` (dark). No Tailwind `dark:` prefix anywhere.

```css
/* Surface layers */
--color-bg          /* page background */
--color-surface     /* card / panel background */
--color-surface-raised  /* elevated surface (e.g. dropdown) */
--color-border      /* subtle dividers */

/* Text */
--color-text        /* primary text */
--color-text-muted  /* secondary / metadata */
--color-text-faint  /* placeholder / disabled */

/* Accent (violet) */
--color-accent      /* interactive elements, focus rings */
--color-accent-muted /* hover states, subtle highlights */

/* Status colours (muted) */
--color-status-spike
--color-status-resolved
--color-status-blocked
--color-status-ready
--color-status-doing
--color-status-done
```

### Typography
- **UI text**: System sans-serif stack (`system-ui, -apple-system, sans-serif`)
- **Ticket IDs, code**: Monospace (`ui-monospace, 'Cascadia Code', monospace`)
- **Scale**: Keep it tight — 13–14px for dense UI, 15–16px for readable body text

### Accessibility
Best effort: reasonable contrast, visible focus states on all interactive elements, no colour-only communication (always pair colour with icon or label for status).
